# BUILD FEATURE: ECHO GUIDED SETUP WIZARD

## PROJECT OVERVIEW

Build an intelligent, interactive onboarding and integration setup wizard inside the Echo platform.

The goal is to make connecting external accounts and services extremely simple for non-technical business owners.

The user should not need to understand API keys, developer settings, webhooks, access tokens, permissions, or complicated third-party dashboards.

The experience should feel similar to DocuSign, where an indicator moves from one required action to the next and clearly shows the user where to click.

Echo should guide the customer through every setup step using:

* Spoken or written instructions
* Highlighted buttons
* Moving arrows or tabs
* Screen overlays
* Progress tracking
* Contextual help
* Screenshot-based screen analysis when the user gets stuck
* A self-updating guidance library that improves when third-party platforms change their interfaces

This feature should become part of Echo’s main customer onboarding system.

---

# PRIMARY OBJECTIVE

Create an onboarding experience where a customer can:

1. Create an Echo account.
2. Answer a short list of questions about their business.
3. Choose which services they want to connect.
4. Click “Connect.”
5. Log into the selected service.
6. Follow Echo’s highlighted instructions.
7. Complete the setup without needing technical knowledge or outside help.

The customer experience should feel like:

> “Echo is sitting beside me and showing me exactly what to do.”

# DESIGN PRINCIPLES

The entire wizard must follow the KISS principle:

## Keep It Simple

The interface should:

* Use plain English.
* Show one action at a time.
* Avoid technical terminology.
* Avoid large forms.
* Avoid overwhelming the user with too many options.
* Clearly explain why each permission is needed.
* Allow users to skip optional integrations.
* Save progress automatically.
* Allow the user to leave and continue later.
* Work well on desktop, tablet, and mobile.
* Use large, clear buttons.
* Provide immediate confirmation after each completed step.

Do not display API keys, client secrets, developer settings, webhook URLs, or technical configuration details to normal customers unless absolutely required.

Whenever possible, use official OAuth login and authorization flows.

# GUIDED SCREEN OVERLAY

Build a screen guidance layer that can display instructions on top of supported setup pages.

The guidance layer should include:

* A highlighted border around the next button or field
* A moving arrow or pointer
* A floating instruction tab
* A dimmed background outside the highlighted area
* A short instruction
* A Next button when appropriate
* A Back button
* A Repeat Instruction button
* A Help Me button
* An Exit Guide button

Example instruction:

> Click “Continue” to allow Echo to connect to your Facebook Page.

The pointer should attach to a detected webpage element, not just a fixed screen coordinate.

Preferred targeting methods:

* DOM selector
* Element ID
* Element role
* Button label
* Accessibility label
* Text content
* Relative element position
* Computer vision fallback

Avoid relying only on x/y coordinates because third-party websites frequently change layouts and screen sizes.

The system should attempt to identify the customer’s current screen before displaying guidance.

Each supported screen should have a stored screen definition.

A screen definition may include:

* Platform name
* Screen name
* Page URL pattern
* Visible page title
* Required text
* Optional text
* Expected buttons
* Expected fields
* DOM selectors
* Accessibility labels
* Reference screenshot
* Device type
* Browser type
* Account type
* Screen version
* Date last verified

Example screen definition:

```json
{
  "platform": "facebook",
  "screen_name": "page_permission_selection",
  "url_patterns": [
    "facebook.com/dialog/oauth",
    "facebook.com/v*/dialog/oauth"
  ],
  "required_text": [
    "Choose what you allow"
  ],
  "target_action": {
    "type": "button",
    "text": "Continue"
  },
  "version": "2026-01",
  "status": "verified"
}
```

# NORMAL GUIDANCE PROCESS

The guidance engine should follow this logic:

1. Detect the current page.
2. Match it against known screen definitions.
3. Identify the next required action.
4. Highlight the correct element.
5. Display a simple instruction.
6. Detect when the action is completed.
7. Move automatically to the next step.
8. Update the setup progress.
9. Confirm successful connection.

---

# SCREENSHOT ANALYSIS FLOW

When the user approves screen analysis:

1. Capture the active setup screen.
2. Automatically blur or mask obvious sensitive information when possible.
3. Send the screenshot to the screen-analysis service.
4. Identify:

   * Platform
   * Screen type
   * Visible buttons
   * Fields
   * Warnings
   * Error messages
   * Permission options
   * Likely next action
5. Return the location and label of the recommended action.
6. Move the guidance tab or highlight to that location.
7. Explain the next step in plain English.
8. Ask the customer to confirm if the suggested element is correct when confidence is low.

Example:

> Facebook has changed this page. Click the blue “Continue” button in the lower-right section of the window.

---

# CONFIDENCE LEVELS

Every screen-analysis result should include a confidence score.

Example:

```json
{
  "platform": "facebook",
  "screen_type": "permission_review",
  "recommended_action": "Click Continue",
  "target_text": "Continue",
  "confidence": 0.94
}
```

Behavior:

* Confidence above 0.90: highlight automatically.
* Confidence between 0.70 and 0.89: highlight and ask for confirmation.
* Confidence below 0.70: do not guess. Offer alternate instructions or human support.

Never automatically click financial, legal, publishing, deletion, account-management, or permission-sensitive buttons on behalf of the user without explicit approval.

---

# SELF-UPDATING GUIDANCE LIBRARY

When Echo discovers that a third-party platform has changed its interface, it should create a proposed new screen version.

Do not immediately replace the existing verified guide based on one screenshot.

New discoveries should move through these statuses:

* Unrecognized
* Proposed
* Under Review
* Verified
* Active
* Deprecated
* Rejected

When a new screen is detected:

1. Save a sanitized representation of the screen.
2. Save detected text and page structure.
3. Save the recommended action.
4. Save the integration and setup step.
5. Save device and browser information.
6. Create a proposed screen definition.
7. Compare it with existing screen versions.
8. Determine whether it is:

   * A completely new screen
   * A new layout variation
   * A mobile version
   * A business-account variation
   * A regional variation
   * A temporary error screen
9. Send it to an admin verification queue.
10. After verification, activate the new guide for matching future users.

---

# LEARNING RULE

Echo should learn from interface changes once, then reuse that knowledge for future customers.

However, it must not assume every customer sees the same layout.

Store multiple verified variations for:

* Desktop
* Mobile
* Tablet
* Different browsers
* Personal accounts
* Business accounts
* Facebook Pages
* Business Manager accounts
* New and old interface versions
* Regional interface differences
* Permission variations
* Error screens

The system should choose the correct guide based on the user’s current screen.

# PRIVACY AND SECURITY REQUIREMENTS

This feature may handle sensitive screens and account permissions.

Implement the following protections:

* Require user permission before every screen capture unless the user has explicitly enabled temporary session-based permission.
* Never capture passwords.
* Never capture payment information.
* Attempt to detect and mask:

  * Password fields
  * Credit card fields
  * Social Security numbers
  * Bank information
  * Access tokens
  * API secrets
  * Private messages
  * Personal contact information
* Encrypt screenshots during transmission and storage.
* Automatically delete raw screenshots after analysis unless the user agrees to contribute the sanitized version to the improvement system.
* Store sanitized screenshots separately from raw screenshots.
* Limit admin access to authorized personnel.
* Log all admin screenshot access.
* Allow customers to delete their setup screenshots.
* Clearly explain what is captured and why.
* Never use customer screenshots for unrelated model training without explicit permission.
* Never allow one customer’s private data to appear in another customer’s guidance.

# ERROR HANDLING

The setup wizard must recognize common problems.

Examples:

* User is logged into the wrong account.
* User does not own a Facebook Page.
* User is not a page administrator.
* Required permission was denied.
* Browser blocked the pop-up.
* OAuth window was closed.
* Session expired.
* Connection token expired.
* Facebook or Google returned an error.
* Account requires two-factor authentication.
* Business account verification is incomplete.
* User selected the wrong business.
* User has multiple pages.
* Connection succeeded but verification failed.

For each error, display:

* What happened
* Why it may have happened
* The simplest next action
* Try Again button
* Help Me button
* Skip for Now button
* Contact Support button

Do not display raw developer error messages to customers.

Log technical errors for administrators.

# ACCEPTANCE CRITERIA

The feature is considered successful when:

1. A non-technical user can complete the main Echo setup without developer assistance.
2. Users are never required to manually create API keys for supported OAuth integrations.
3. The wizard displays one clear action at a time.
4. Known screens are recognized automatically.
5. The guide correctly highlights the next action.
6. Users can click Help Me when a screen changes.
7. Screenshots are only captured with permission.
8. The system can propose a new screen guide when a platform changes.
9. New guides require validation before becoming active.
10. Future users can benefit from previously verified screen changes.
11. Setup progress is saved automatically.
12. Customers can leave and continue later.
13. Admins can review failures and new screen discoveries.
14. Sensitive information is protected.
15. The system is reusable across multiple integrations.
