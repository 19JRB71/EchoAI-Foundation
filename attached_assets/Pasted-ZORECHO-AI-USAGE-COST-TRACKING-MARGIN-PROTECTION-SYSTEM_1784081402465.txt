Important context

We are currently developing an Echo Orchestrator prototype.

That prototype is intended to let Echo:

Reason naturally instead of relying only on fixed commands
Answer general questions
Understand the complete Zorecho platform
Navigate users to the correct location
Choose the appropriate tool or agent
Delegate tasks to Zorecho’s specialist agents
Use live services when current information is required

The usage system must include this prototype from the beginning.

Do not build a separate cost meter that ignores the prototype.

Every model call, tool call, delegated action, voice request, and agent execution initiated through the Echo Orchestrator must be recorded.

Echo Orchestrator tracking

Every Echo Orchestrator request must have a unique request or trace ID.

The system must track the full chain, including:

User message
Intent classification
Model used for reasoning
Tools selected
Agents delegated to
Additional AI calls made by those agents
External API calls
Retries
Final response
Total estimated cost of the complete interaction

Example:

User asks Echo to create a Facebook campaign
→ Echo reasoning call
→ Scout research call
→ Forge image generation
→ Atlas campaign preparation
→ Echo final summary

The owner dashboard should show the cost of the entire workflow and the cost of each component inside it.

This is necessary so a single request cannot appear inexpensive while secretly generating multiple downstream calls.

Internally estimated cost
Provider-reported cost

Use the provider-reported value as the final reconciled value when reliable.

Provider reconciliation

Where a provider offers usage or billing APIs:

Pull provider usage on a scheduled basis
Reconcile it with Zorecho’s internal records
Flag unexplained differences
Identify unassigned usage
Show reconciliation status
Never silently overwrite the original internal estimate

Where a provider does not expose usable cost information:

Calculate the estimate from recorded usage units
Clearly mark it as an estimate
Allow an administrator to enter or update pricing manually

The system must continue working even when a provider’s usage API is temporarily unavailable.

Cost-aware routing

The Echo Orchestrator should use the least expensive method that can reliably complete the request.

Examples:

Navigation request → no large model
Known platform help → cached answer or retrieval
Simple classification → inexpensive model
Basic summary → lower-cost model
Complex strategy → advanced reasoning model
Live information → appropriate tool plus concise model response
Repeated identical request → cache when appropriate

Do not reduce quality blindly.

The routing system should consider:

Complexity
Required accuracy
Live-data requirements
Customer’s subscribed intelligence tier
Remaining included capacity
Safety and permission requirements

Record why the orchestrator selected a particular model or tool.

Caching and duplication prevention

Identify safe opportunities to reduce repeated cost:

Feature help
Onboarding explanations
Stable business profiles
Previously generated brand information
Repeated industry facts
Repeated reports with unchanged source data
Duplicate background research
Identical user requests within a short period

Do not cache:

Private data across customers
Time-sensitive information beyond an appropriate expiration
Actions that must execute again
Responses whose underlying data changed

Every cache hit should be recorded as avoided estimated cost.

Demo and private-test usage

We are currently in private testing.

The usage system must distinguish between:

Development usage
Internal private testing
Demo usage
Customer production usage

Demo and owner testing must not be assigned to a paying customer.

However, it should still be measured so we understand the real cost of demos and internal use.

Security and privacy
Never log raw API keys
Avoid storing unnecessary sensitive prompt content
Use role-based access
Customers may only view their own usage
Owner and approved admins may view company-wide economics
Preserve tenant isolation
Audit access to financial usage information
Encrypt sensitive data where appropriate
Define retention rules for detailed request logs
Reliability requirements

Usage logging must not cause the primary customer action to fail.

If usage recording temporarily fails:

Queue the record for retry
Preserve the original request result
Alert the owner if records remain unreconciled
Never silently discard usage

Use idempotency so retries do not create duplicate charges.
