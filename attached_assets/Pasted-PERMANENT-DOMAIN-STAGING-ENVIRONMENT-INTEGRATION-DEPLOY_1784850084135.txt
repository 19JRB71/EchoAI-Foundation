PERMANENT DOMAIN, STAGING ENVIRONMENT & INTEGRATION DEPLOYMENT PLAN

Now that I own zorecho.com, I want to stop using temporary Replit preview URLs as the primary testing environment. From this point forward, we are transitioning Zorecho from a development project into a professionally managed application with dedicated environments for development, staging, and production.

This is now the highest-priority infrastructure task. No new feature development should take precedence over completing this transition.

Objective

Establish a permanent staging environment using my domain so every external integration (Google, Facebook, Instagram, Stripe, Twilio, etc.) is configured against stable URLs rather than temporary preview domains.

Replit preview URLs should only be used for development and debugging—not for validating customer-facing integrations.

OAuth & External Integrations

All OAuth providers must be updated to use permanent callback URLs instead of temporary Replit preview domains.

Examples:

Google

https://staging.zorecho.com/api/google/oauth/callback

Facebook

https://staging.zorecho.com/api/facebook/oauth/callback

Instagram

Uses the Facebook OAuth callback.

Future providers should follow this same permanent structure.

No customer-facing integration should rely on temporary development URLs.

Customer Experience Standard

The application should never allow a customer to begin a connection that the system already knows cannot succeed.

If an integration is unavailable, incomplete, or missing configuration:

Do not display an active Connect button.

Instead display:

Setup Required

Explain:

Why it is unavailable
What configuration is missing
What action is required
Whether the owner or the customer needs to complete that action

Customers should never be redirected to unexplained provider error pages whenever the application can determine beforehand that the connection will fail.

If an unexpected provider error occurs, return the customer to Zorecho with a clear explanation and recovery options.

Permanent Development Rule

No Green Button Without a Green Backend.

Before enabling any customer-facing integration, the system must verify:

Required secrets exist
Environment variables are present
Redirect URIs match
Provider application is configured
Required permissions exist
Backend services are operational

Only then should the integration appear available.
